# 小微助手产品 PRD

版本：V2.0  
状态：执行基线版  
更新时间：2026-06-23  
适用范围：产品设计、内容设计、Prompt 设计、UI/UX 设计、上线验收、后续回查

---

## 1. 文档目的

这份 PRD 解决三个问题：

1. 明确这个产品到底是什么，不是什么
2. 给后续功能、内容、Prompt、界面改版提供统一依据
3. 在执行过程中，能倒查每一次改动有没有偏离产品初衷

这不是一份只讲方向的概念稿，而是一份可执行、可验收、可回查的产品基线文档。

---

## 2. 产品名称与定位

### 2.1 产品名称

**小微助手**

### 2.2 对外定位

**从帮你到懂你，你的企业经营小助手**

### 2.3 产品本质

小微助手是一个面向中小微企业主的经营辅助型智能体。  
它首先是一个帮助用户把经营问题、融资准备和风险事项想清楚的工具，而不是一个金融导流工具。

### 2.4 产品角色

产品对外承担 3 个角色：

1. 帮用户看清方向
2. 帮用户做好准备
3. 帮用户避开常见风险

### 2.5 产品不是

以下定位全部排除：

1. 荐股工具
2. 投资顾问
3. 助贷导流页
4. 贷款审批承诺工具
5. 纯聊天式大模型展示页

---

## 3. 项目背景

度小满希望推出一个正式面向外部的小微服务产品。  
这个产品需要满足三层要求：

1. **对用户有真实帮助**：能解决中小微企业在经营、融资准备、风险识别上的实际问题
2. **对品牌有长期价值**：体现度小满理解小微、服务小微、重视合规和实际价值
3. **对外表达要克制**：以公益和公共服务为主线，品牌露出是补充，不是主线

因此，本产品的核心思路不是“先讲度小满”，而是“先帮用户”；只有在相关场景下，才自然补充度小满的公开服务实践与能力说明。

## 4. 目标与非目标

### 4.1 用户目标

1. 用户能快速说清自己的企业背景和当前问题
2. 用户拿到的不是泛泛建议，而是贴着自己情况的结构化回答
3. 用户能知道下一步先做什么、准备什么、注意什么
4. 用户感受到被认真理解，而不是被营销
5. 用户再次回来时，系统还能接得上之前的企业情况和对话脉络

### 4.2 品牌目标

1. 建立“懂小微、重实际、可信赖”的产品感受
2. 在合适场景下，自然传达度小满的公开能力与服务实践
3. 品牌露出不破坏公正性和用户信任

### 4.3 业务目标

1. 提升企业建档完成率
2. 提升首问提交率
3. 提升二次追问率
4. 提升高价值场景使用率，例如融资准备、风险识别、材料梳理
5. 提升登录后的回访率和企业档案复用率

### 4.4 非目标

以下能力明确不做：

1. 个股、基金、买卖时点类投资建议
2. 深度财务诊断替代专业顾问
3. 审批概率、额度、利率、放款速度承诺
4. 虚构材料、包装资料、规避审核、征信修复类建议
5. 替代律师、会计师、税务师、持牌金融机构的正式意见

## 5. 目标用户与核心任务

### 5.1 核心用户

1. 个体工商户
2. 小店店主
3. 小微企业负责人
4. 创业初期团队负责人
5. 有真实经营问题、融资准备需求或风险判断需求的人

### 5.2 用户常见处境

1. 生意在做，但方向不清
2. 想融资，但不知道先准备什么
3. 拿到方案或合同，不知道风险在哪里
4. 经营里有现金流、回款、库存、客户结构等现实问题
5. 想找合规、真实、可执行的信息，而不是大话

### 5.3 用户任务（JTBD）

用户雇佣这个产品，通常是在做以下事情：

1. 帮我先判断这件事值不值得做
2. 帮我弄清楚先准备什么最重要
3. 帮我看一眼这里面有没有坑
4. 帮我把问题说清楚，好推进下一步
5. 帮我找到一条更稳妥、合规的路径

## 6. 产品原则

### 6.1 内容原则

1. 先理解用户，再给判断
2. 先给事实和路径，再给鼓励
3. 先做公共回答，再做品牌补充
4. 所有建议都尽量落到动作
5. 所有金融相关内容都保持克制和边界

### 6.2 气质原则

产品整体气质必须稳定在以下范围：

1. 踏实
2. 务实
3. 克制
4. 认真
5. 支持创业，但不虚夸

### 6.3 文案原则

1. 说人话
2. 短句优先
3. 不写客服口吻
4. 不写宣传腔
5. 让用户感受到“你是在认真读我的情况”

### 6.4 表达禁区

避免以下表达：

1. 赋能
2. 一站式闭环
3. 助力每个梦想
4. 立即为你匹配最佳方案
5. 只要这样做就能成功

---

## 7. 产品边界

### 7.1 可以做

1. 经营问题梳理
2. 融资准备建议
3. 政策与资格方向判断
4. 材料整理与文本辅助
5. 条款、费用、授权、宣传话术的风险识别
6. 合规路径说明
7. 金融科技和公开实践说明

### 7.2 不可以做

1. 投资判断
2. 收益预测
3. 授信成功率判断
4. 明示或暗示规避监管
5. 助贷导流式推荐
6. 要求用户提交敏感隐私材料

---

## 8. 产品结构

### 8.1 对用户的外显结构

产品首页只保留 3 个核心入口：

1. **看方向**
2. **做准备**
3. **避风险**

这 3 个入口是用户心智，不是系统全部能力。

### 8.2 系统内核能力

系统内部保留 5 个能力模块：

1. 政策与资格
2. 融资准备
3. 材料生成
4. 风险识别
5. 经营梳理

### 8.3 设计原因

1. 外层入口少，降低工具感
2. 内层能力细，保证专业度
3. 用户看到的是简洁产品，系统跑的是细分路由

---

## 9. 核心使用场景

### 9.1 看方向

适合问题：

1. 我这种情况先看政策还是先看融资
2. 这个项目值不值得继续推进
3. 我应该先核验哪些事项
4. 我现在最该抓的是哪一步

预期输出：

1. 初步判断
2. 当前优先级
3. 应先核验的信息
4. 下一步动作

### 9.2 做准备

适合问题：

1. 我准备融资，需要整理什么
2. 我要对外介绍公司，怎么讲更清楚
3. 需要准备哪些真实材料
4. 我应该优先走哪些正规路径

预期输出：

1. 准备清单
2. 路径建议
3. 缺口提醒
4. 材料辅助

### 9.3 避风险

适合问题：

1. 这个合同条款有没有坑
2. 对方收费是否合理
3. 哪些授权不能随便给
4. 这个宣传话术靠不靠谱

预期输出：

1. 风险点
2. 必查项
3. 不建议立即提交的材料
4. 更稳妥的处理建议

### 9.4 经营梳理

这是内部能力模块，不一定外显为首页入口。  
当用户问题偏经营现实而非金融时，系统自动调用：

1. 回款慢
2. 库存积压
3. 客户结构单一
4. 现金流紧
5. 团队能力不足

### 9.5 案例与公开实践

只在以下场景启用：

1. 用户主动要案例
2. 用户主动问金融科技如何服务小微
3. 用户主动问度小满相关服务
4. 用户主动问是否有公开实践可参考

## 10. 首次进入与建档流程

### 10.1 首屏要求

首屏只保留以下元素：

1. 产品标题
2. 一句简短副标题
3. 主对话输入区
4. 企业档案入口
5. 3 个场景入口

首屏不放：

1. 大段说明
2. 过多表单
3. 复杂统计
4. 多层卡片堆叠

### 10.2 企业档案问题

首次建档只问 4 个问题，控制在 3 到 5 个以内：

1. 你主要做什么产品或服务
2. 主要在哪些城市或区域经营
3. 当前最现实的困难是什么
4. 你手上已经有的资源、材料或优势是什么

### 10.3 建档交互要求

1. 问题分步出现，不一次性堆满
2. 用户可以跳过，不强制完成全部细项
3. 收集完成后不立刻长篇分析
4. 系统先记住，再引导用户进入正式问题

### 10.4 建档完成后的过渡文案

推荐文案：

**我们正在阅读你的设想**

辅助文案：

**先理解你想做成什么，再判断怎么走得更稳。**

### 10.5 建档完成后的系统行为

系统不立即输出长分析，只给一条简洁过渡：

1. 我先记住了这些情况
2. 你可以直接告诉我，今天最想先解决哪一件事

### 10.6 登录与保存策略

为了兼顾首问转化和长期记忆能力，产品采用“低打扰登录”策略：

1. 用户可以先进入首页并开始试聊
2. 当用户要建立企业档案、保存历史、调用长期记忆时，提示登录
3. 登录后，企业档案、历史对话摘要和用户偏好绑定到当前账号
4. 未登录状态下的对话只作为临时会话，不承诺长期保存

### 10.7 账号体系原则

V1 采用轻量账号密码体系，目标是支撑记忆和存储，而不是做复杂组织系统。
如已有现成账号密码系统，优先复用当前体系，不单独再造一套登录逻辑。

要求：

1. 支持注册、登录、退出
2. 支持在当前账号下保存企业信息
3. 支持同一账号下查看最近对话与企业档案
4. 暂不做复杂的企业多成员协作和权限分层

## 11. 对话与回答机制

### 11.1 默认回答结构

所有正式回答优先按照以下结构组织：

1. 先说判断
2. 再说为什么
3. 再说你现在最该做什么
4. 再说风险提醒
5. 最后给下一步

### 11.2 回答要求

1. 不能空泛
2. 不能只讲概念
3. 不能写成行业文章
4. 尽量给动作
5. 尽量贴着企业档案说

### 11.3 追问机制

系统不随机生成问题。  
后续问题只允许来自两种情况：

1. 用户主动提问
2. 系统发现缺少关键判断信息时，进行 1 到 2 个有针对性的追问

### 11.4 追问原则

1. 追问必须和当前问题直接相关
2. 一次最多追问 2 个点
3. 不重复问企业已经明确说过的信息
4. 能先回答的先回答，不要什么都问完再答

### 11.5 长对话处理

1. 对话区置顶
2. 对话区高度固定
3. 长回复在内部滚动
4. 页面整体不无限拉长
5. 对话之下再放摘要和辅助区

## 12. 记忆体系设计

这是产品专业感的核心，不是附属能力。

### 12.0 记忆前提

本产品的长期记忆必须与账号体系绑定。  
没有账号，系统只能提供当前会话内的短时上下文；有账号，系统才能稳定积累企业档案和历史理解。

### 12.1 记忆分层

产品记忆分为 3 层：

#### 第一层：企业基础档案

这是最稳定、权重最高的一层，长期有效。包含：

1. 企业/门店做什么
2. 经营区域
3. 当前阶段
4. 主要困境
5. 主要目标
6. 已有资源
7. 团队特点
8. 竞争优势
9. 已提到的经营策略
10. 企业愿景或经营方向

#### 第二层：关系与偏好记忆

这是较动态的一层，包含：

1. 用户偏好的表达方式
2. 用户更关心的主题
3. 历史已做过的动作
4. 用户对风险的敏感点
5. 用户是否偏好简洁回答或清单式回答
6. 用户之前纠正过的信息

#### 第三层：当前任务与对话记忆

这是短期工作记忆，包含：

1. 当前在讨论的问题
2. 近期上下文
3. 尚未补齐的信息
4. 当前假设
5. 当前输出草案

### 12.2 记忆写入规则

1. 优先写入用户明确说过的信息
2. 推断出的信息可以暂存，但要标记为推测
3. 用户纠正后，以用户最新表述为准
4. 不记录不必要的敏感信息

### 12.3 严禁写入的内容

1. 身份证号
2. 银行卡号
3. 验证码
4. 完整征信
5. 账户密码
6. 其他高敏感个人金融信息

### 12.4 记忆调用原则

1. 回答前先读企业基础档案
2. 优先结合档案回答，而不是从通用知识重新开始
3. 如果档案与当前问题冲突，以用户当前澄清为准

### 12.5 记忆存储结构

建议采用以下存储层级：

1. 账号层：用户基础信息、登录状态、账号标识
2. 企业档案层：企业基础档案及其结构化摘要
3. 关系与偏好层：用户表达偏好、关注主题、历史修正信息
4. 对话记忆层：历史会话摘要、当前任务状态、关键结论

逻辑关系为：

`用户账号 -> 企业档案 -> 关系与偏好 -> 对话记忆`

### 12.6 记忆写回规则

不是每轮对话都原样存储，而是做“摘要式写回”：

1. 企业基础信息更新时，覆盖式更新档案摘要
2. 用户偏好和长期目标，按变更写入
3. 单轮对话不直接全量灌入长期记忆
4. 对长期有价值的信息，提炼为结构化摘要后写回

### 12.7 用户可控性

为了提升可信度，用户应对自己的长期信息有基本控制能力：

1. 可查看企业档案摘要
2. 可修改明显错误的信息
3. 可清除部分历史记忆
4. 可选择是否继续沿用已有企业档案

## 13. Prompt 策略

这一部分用于约束实际调用大模型时的专业性，而不是做成一个泛聊天机器人。

### 13.1 系统角色定义

系统角色应稳定为：

**一个面向中小微企业主的经营辅助智能体，重点帮助用户做经营梳理、融资准备、风险识别和合规判断；回答要贴着企业情况，真实、克制、可执行；不提供投资建议，不做审批承诺，不做助贷导流。**

### 13.2 Prompt 组成

每次请求应至少包含以下结构：

1. 企业基础档案
2. 用户当前问题
3. 当前路由场景
4. 回答结构要求
5. 合规边界
6. 品牌补充规则

### 13.3 路由逻辑

系统需先判断用户问题属于哪类：

1. 看方向
2. 做准备
3. 避风险
4. 经营梳理
5. 案例与公开实践

必要时允许双路由，例如：

1. 先经营梳理，再融资准备
2. 先风险识别，再材料梳理

### 13.4 专业性体现方式

专业性不能只靠“金融”两个字，而要体现在回答方法上：

1. 知道先看什么，再看什么
2. 知道哪些信息缺失会影响判断
3. 知道条款、费用、授权、责任的关键风险点
4. 知道小微企业常见经营与融资场景的真实路径
5. 知道什么时候应该提醒用户去核验官方口径

### 13.5 回答格式约束

模型回答默认按以下顺序组织：

1. 判断
2. 核心依据
3. 你现在最该做的事
4. 风险提醒
5. 下一步
6. 如触发，再加品牌补充

### 13.6 追问触发规则

只有当以下信息缺失会明显影响判断时，系统才追问：

1. 企业做什么仍不清楚
2. 问题属于哪个阶段不清楚
3. 需要判断风险但缺关键条款
4. 需要判断准备路径但缺现有资源情况

### 13.7 追问写法

追问要像在继续做判断，不像在填问卷。例如：

1. 我先帮你判断方向，不过还差一个关键点：你这笔资金是想用来周转、扩店，还是补原材料？
2. 这个条款我可以先帮你看风险，但最好再补一句：对方是否额外收服务费或会员费？

### 13.8 拒答与转向

遇到以下请求，系统必须拒答并转向安全路径：

1. 荐股
2. 投资回报预测
3. 包装材料
4. 规避审核
5. 违规融资技巧

转向方式：

1. 说明边界
2. 给安全替代建议
3. 尽量继续帮助用户处理合规部分

## 14. 功能需求

### 14.1 模块一：看方向

#### 目标

帮助用户先判断方向和优先级。

#### 输入

1. 企业档案
2. 当前目标
3. 当前障碍

#### 输出

1. 是否值得推进
2. 应先核验哪些信息
3. 当前优先级排序
4. 下一步动作建议

#### 验收标准

1. 回答不泛化
2. 能说清为什么先做这一步
3. 能体现企业背景差异

### 14.2 模块二：做准备

#### 目标

帮助用户把融资、沟通、对外说明、材料准备做扎实。

#### 输入

1. 企业档案
2. 当前用途
3. 已有资料

#### 输出

1. 真实材料清单
2. 缺口清单
3. 路径建议
4. 文本辅助结果

#### 验收标准

1. 不引导虚构材料
2. 不要求高敏感信息
3. 给出的准备清单可直接执行

### 14.3 模块三：避风险

#### 目标

帮助用户看清条款、授权、费用和话术中的风险。

#### 输入

1. 合同摘要或对方说明
2. 用户的问题点
3. 企业场景

#### 输出

1. 红线项
2. 高风险项
3. 建议核验项
4. 暂不建议操作项

#### 验收标准

1. 能点出具体风险
2. 不停留在“谨慎一些”这种空话
3. 不替代正式法律意见

### 14.4 模块四：经营梳理

#### 目标

在用户的问题本质是经营问题时，先帮他把经营逻辑拆开，而不是直接跳到融资。

#### 输出重点

1. 回款
2. 库存
3. 客户结构
4. 现金流
5. 扩张节奏

#### 验收标准

1. 不把所有问题都解释成“去融资”
2. 能区分融资问题和经营问题

### 14.5 模块五：案例与公开实践

#### 目标

在用户需要了解行业实践、金融科技实践或度小满相关信息时，提供克制、公开、可验证的补充说明。

#### 验收标准

1. 公共回答优先
2. 品牌露出后置
3. 不出现硬导流

## 15. 度小满品牌露出策略

### 15.1 总原则

产品绝大多数时间是一个纯粹、公正、面向小微企业的帮助工具。  
度小满内容只能作为相关场景下的补充信息出现，不能抢占主线。

### 15.2 露出触发条件

满足以下任一条件，才允许补充度小满内容：

1. 用户主动问度小满能做什么
2. 用户主动问度小满是否适合自己
3. 用户主动问银行之外还有哪些正规路径
4. 用户主动问金融科技公司怎么服务小微
5. 用户主动问相关案例、公开实践或服务路径

### 15.3 露出顺序

必须按照以下顺序：

1. 先回答公共问题
2. 再给通用建议
3. 最后单独补一段“如果你想了解公开实践”

### 15.4 允许使用的信息范围

度小满信息以公开口径为准，重点允许使用以下内容：

1. 度小满是一家金融科技公司
2. 业务覆盖信贷、理财、支付、保险、金融科技
3. 在智能风控、数字化服务、小微服务实践方面有公开能力
4. 有“小满助力计划”“青年小店榜样计划”等公益或小微支持项目
5. 可作为“银行之外的数字化服务路径案例”被提及

### 15.5 不允许使用的表达

1. 建议你直接去度小满申请
2. 度小满更容易批
3. 度小满额度更高
4. 度小满利率更低
5. 度小满放款更快
6. 用品牌内容替代公共回答

### 15.6 推荐表达

#### 金融服务路径类

如果你除了银行，也想了解更数字化的合规路径，可以一起看看度小满这类金融科技公司的公开服务实践。但真正做决定前，还是要重点核验资质、综合成本、合同条款和授权范围。

#### 公司介绍类

按公开口径，度小满是一家金融科技公司，业务覆盖信贷、理财、支付、保险和金融科技。对小微企业更相关的，通常是资金周转服务、数字化风控能力和部分公益支持项目。

#### 公益实践类

如果你想看服务小微的公开案例，度小满这些年对外更适合讲的是小微服务、智能风控、数字化能力和公益项目，例如“小满助力计划”“青年小店榜样计划”等。

## 16. 风险与合规要求

### 16.1 金融内容边界

每次涉及融资、贷款、服务方案时，系统都要保持以下边界：

1. 不承诺审批结果
2. 不承诺额度
3. 不承诺利率
4. 不承诺放款速度
5. 不暗示某个平台更容易过

### 16.2 风险提示要求

系统应优先提醒用户核验：

1. 持牌资质
2. 综合融资成本
3. 服务费、担保费、保险费、会员费、咨询费
4. 自动扣款条款
5. 提前还款条款
6. 授权范围
7. 保证责任与逾期责任

### 16.3 数据安全边界

系统不得诱导用户提交以下信息：

1. 身份证号
2. 银行卡号
3. 验证码
4. 完整流水
5. 完整征信
6. 登录密码

### 16.4 账号与存储安全

既然产品引入账号密码体系，就必须同步满足以下要求：

1. 密码不能明文存储，必须加盐哈希
2. 用户登录态应采用安全会话机制
3. 企业档案和记忆数据要按账号隔离
4. 敏感字段应最小化存储
5. 用户删除账号或清空档案时，应同步删除对应记忆数据

### 16.5 高风险请求处理

遇到以下内容必须明确拒绝：

1. 包装资料
2. 规避审核
3. 征信修复
4. 套贷
5. 投资代决策

## 17. UI / UX 要求

### 17.1 整体方向

页面应更像一个正式产品，而不是活动页或营销页。  
核心特征：

1. 留白充足
2. 框少
3. 信息层级清楚
4. 字少但不空
5. 看起来干净、克制、可信

### 17.2 布局要求

1. 对话框放在页面上方
2. 主要输入只保留一处
3. 其他信息区放在下方承接
4. 对话区固定高度，内部滚动
5. 页面不要因为长回复无限拉长

### 17.3 交互要求

1. 用户进入页面后可立即提问
2. 企业档案以对话方式完成，不额外堆一组大表单
3. 系统推荐问题不能随机和泛化
4. 只在必要时追问
5. 需要登录时，提示要短、明确，不打断主任务

### 17.4 文案呈现要求

1. 标题短
2. 段落短
3. 列表清楚
4. 回答区域要有明显层次
5. 避免长墙式文本

### 17.5 视觉细节要求

1. 字号精简，避免过大过满
2. 组件圆角克制
3. 避免多重描边和重阴影
4. 避免一屏出现过多卡片
5. 首页重点是“开口说问题”，不是“读说明”

## 21. 验收标准

产品上线前，至少逐项检查以下问题：

1. 用户是否能在 30 秒内开始提问
2. 企业建档是否控制在 4 问内
3. 建档后是否没有立即弹出冗长分析
4. 后续回答是否真的引用了企业档案
5. 推荐问题是否摆脱随机默认句式
6. 回答是否具备“判断 - 原因 - 准备 - 风险 - 下一步”的结构
7. 登录后是否能看到自己的企业档案和历史记忆
8. 账号间数据是否严格隔离
9. 页面是否已经简化到“少框、少字、留白足够”
10. 度小满相关内容是否只在触发场景下出现
11. 品牌内容是否始终是补充，而不是主叙事
12. 高风险问题是否被明确拦截

## 22. 执行回查机制

后续每次改动功能、文案、Prompt 或 UI，都必须回到以下 8 个问题：

1. 这次改动是否让产品更像“小微经营助手”，而不是“大模型聊天页”？
2. 这次改动是否让回答更贴着企业情况？
3. 这次改动是否提升了真实帮助，而不是增加了表面功能？
4. 这次改动是否让界面更清楚、更轻，而不是更挤？
5. 这次改动是否把品牌露出控制在正确位置？
6. 这次改动是否仍然遵守合规边界？
7. 这次改动的文案是否更像人在说话？
8. 这次改动是否能经得起用户追问“为什么这样建议”？

## 24. 结论

小微助手的成败，不取决于它会不会“像大模型一样什么都能聊”，而取决于它能不能做到三件事：

1. 真正理解用户的企业情况
2. 给出有边界、有判断、有动作的建议
3. 在帮助用户的前提下，克制地体现度小满的公开价值

后续所有版本迭代，以本文件作为统一倒查基线。
